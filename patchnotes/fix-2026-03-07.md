# GemmaEvolve Patch Notes - 2026-03-07

## Multi-Checkpoint Benchmark Redesign
- **Fixed Iteration Benchmarking:** Rewrote `run_completion_benchmark.py` and `run_completion_benchmark.sh` to support a multi-checkpoint benchmark design. OpenEvolve now runs for a fixed 800 iterations without early term by default, capturing and recording the best score every 100 iterations (100, 200, ..., 800) into `completion_benchmark.json`.
- **Target Score Early Stopping & Duration Tracking:** If OpenEvolve generates a perfect Verilog module (score >= 1.0) mid-run, it will immediately stop. It now successfully tracks `solved_iteration` and `solved_time_seconds` and outputs them in a formatted final console table.

## Bug Fixes and Stability Improvements
- **Native Gemma Context Adjustments:** Reduced the `n_ctx` parameter for the Native Gemma baseline generation from `131072` down to `32768` for better VRAM safety across the 4x GTX 1080 Ti hardware.
- **OpenEvolve Early Stopping Fix:** Resolved the pervasive issue where OpenEvolve would immediately stop at iteration 1. Removed broken default `--early-stopping-patience 0` flags in the shell script and explicitly set `early_stopping_patience = 99999` in the benchmark Python script configuration to let it run properly.
- **Error Handling on Problem Execution:** The benchmark script now gracefully catches fatal evaluation and pipeline errors (e.g., when the `Llama()` model loading fails). It will now skip the crashing problem and continue rather than bringing down the entire benchmark run.
- **Out of Memory (OOM) GPU Leak Fix:** Fixed a critical VRAM leak across the completion benchmark and OpenEvolve running loops where `llama-cpp-python` models were not being properly released. The scripts now explicitly force C++ cleanup by calling `llm.close()`, eliminating consecutive 'Failed to load model' test skips caused by trailing memory hooks.
- **Segfault on Timeout Fix:** Resolved a segmentation fault occurring in the benchmark when the pure Gemma baseline generation timed out. The previous thread-based timeout continued executing C++ calls in the background while the main thread attempted to release `llm.close()`. Replaced it with a timeout-aware token streaming loop, allowing natural cleanup of the C++ generation loop and preventing OOM / memory corruption.
