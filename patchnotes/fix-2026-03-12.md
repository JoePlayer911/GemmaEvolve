# OpenEvolve Updates - 2026-03-12

## Benchmark Failure Diagnosis & Feedback Improvements
After investigating why 65 out of 141 completion benchmark problems remain unsolved, we identified the root causes and shipped three high-impact fixes.

### 1. Evaluator Error Surfacing in LLM Prompts
**File:** `openevolve/prompt/sampler.py`

Previously, when a Verilog program failed to compile, the evaluator stored the error in the metrics dict — but the LLM prompt system buried it as just another metric value. The LLM had no structured guidance to actually *fix* the compilation error, so it kept regenerating the same broken code (e.g., 749/749 compilation failures on Prob094_gatesv).

Now, `_identify_improvement_areas()` checks for an `error` field in metrics and prominently surfaces it as a **"CRITICAL ERROR"** the LLM must address first. Error messages are truncated at 800 chars to avoid filling the context window.

### 2. Cross-Pollination Enabled Across All Problems
**Files:** 156 × `examples/verilog_eval/*/gemma_config.yaml`

| Setting | Before | After | Effect |
|---------|--------|-------|--------|
| `num_top_programs` | 0 | 2 | LLM sees top 2 programs from other islands for inspiration |
| `num_diverse_programs` | 0 | 1 | LLM sees 1 diverse alternative approach |
| `include_artifacts` | false | true | LLM sees evaluator artifacts (testbench output, execution logs) |

Previously, each island evolved in complete isolation — the LLM only ever saw its own current program. Now islands share their best solutions, enabling cross-pollination of successful strategies.

### 3. Logging Bug Fix — `level must be an integer`
**File:** `openevolve/database.py` (line 1081)

Fixed `logger.log("Warning: Program %s has no embedding...")` → `logger.warning(...)`. The original code passed the log message string as the `level` parameter to `logger.log()`, which expects an integer. This caused a `TypeError` on every iteration encountering a program without an embedding, silently discarding results from islands 2 and 3.
