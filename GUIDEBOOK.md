# GemmaEvolve: LLM Guidebook

This document is technically structured to immediately orient any future Large Language Model (or human engineer) to the architecture, directory structure, and execution flow of the **GemmaEvolve** project. If you are an LLM reading this, consider this your master context file.

---

## 🏗️ 1. Project Architecture & Purpose

**Purpose:** 
GemmaEvolve integrates the **OpenEvolve** evolutionary coding framework with the native **Gemma-3-12B** local model to autonomously solve complex Hardware Design (Verilog) problems via iterative generation, testbench evaluation, and genetic selection (cross-pollination).

**Core Mechanism:**
1. A base LLM (Gemma) generates an initial Verilog module.
2. An `iverilog`-based evaluator (`evaluator.py`) compiles and simulates the code against a testbench.
3. If it fails, the error output is parsed and fed back to the LLM.
4. OpenEvolve creates "Islands" (parallel evolutionary workers) that maintain populations of code. They cross-pollinate successful strategies (via `num_top_programs` config) and iteratively improve fitness (accuracy / combined_score) until a flawless program is born or patience runs out.

## ⚙️ 3. Execution CLI Flags

### The Master Benchmark Runner
`run_completion_benchmark.py` handles the massive parallel execution of all 156 Verilog problems. It *only* triggers OpenEvolve if the native Gemma baseline scores < 1.0.

```bash
python3 run_completion_benchmark.py [OPTIONS]
```
**Key Flags:**
*   `--limit N`: Only test N problems (0 = all).
*   `--iterations N`: Max OpenEvolve generations (Default 800).
*   `--jumpstart`: (Experimental) Seed the evolutionary pool with the exact code generated by the failing baseline.
*   `--fill-gaps`: Auto-detect if `completion_benchmark.json` missed problems and only execute those.
*   `--baseline-only`: Skip OpenEvolve entirely, just test raw zero-shot capability.

### Individual Problem Debugger
If a single problem is failing and you want to debug its evolution in isolation with max verbosity:

```bash
python3 -m openevolve.cli \
    examples/verilog_eval/ProbXXX/initial_program.v \
    examples/verilog_eval/ProbXXX/evaluator.py \
    --config examples/verilog_eval/ProbXXX/gemma_config.yaml \
    --iterations 50 \
    --log-level DEBUG
```
**Key Flags:**
*   `--target-score N`: (Usually 1.0). Stops evolving early when reached.
*   `--checkpoint PATH`: Resume from a specific iteration directory.

## 🛠️ 4. Pro-Tips for Future LLMs Working on this Repo

1. **Debugging "Stuck" Iterations:** 
   If OpenEvolve keeps getting `accuracy: 0.0` repeatedly on a problem, do NOT guess why. Open `logs/evolve_generations/` and read the most recent `.md` file. It will show you exactly what the `iverilog` evaluator returned to the LLM, and why the LLM failed to parse it. 

2. **Benchmark JSON Caching:**
   `run_completion_benchmark.py` gracefully resumes from where it left off by reading `completion_benchmark.json`. If you fix a bug in the evaluator and want to retry a problem that previously scored `0.0`, you MUST manually edit the JSON or delete the problem node from the JSON. Otherwise, the script will say "Already skipped/failed" and ignore it.

3. **VRAM Leaks & llama_cpp_python:**
   If you ever write a script that instantiates `Llama()` or `GGUFLLM()` in a loop, you **must** call `llm.close()` and invoke `gc.collect()`. GGUF models do not naturally release their 12GB+ VRAM footholds, leading to terrifying out-of-memory cascading crashes.

4. **Modifying Evaluator Logic:**
   If you change how an evaluator scores code (e.g. changing `combined_score`), you must update `scripts/agentic_convert.py` (which generates them) AND bulk-update the existing 156 local `evaluator.py` files.

5. **Cross-Pollination Settings:**
   The magic that allows OpenEvolve to actually solve hard problems lives in `gemma_config.yaml`:
   ```yaml
   prompt:
     num_top_programs: 2     # Let islands see the best code of other islands
     num_diverse_programs: 1 # Force the LLM to look at a weird/different approach
     include_artifacts: true # ALWAYS EXPOSE COMPILATION ERRORS TO THE LLM
   ```
